# V2 Planning Notes

## Principle

The product is at the point where it should stop expanding scope and finish Version 1.

The goal is to ship something genuinely useful every day, use it in production for a few weeks, and let real usage decide which Version 2 investments matter most.

## Version 1 definition

### User portal

- [x] Sign up / login
- [x] Upload one or more resumes
- [x] Set country
- [x] Set locations
- [x] Choose companies
- [x] Request a new company
- [x] Choose role families
- [x] Set match threshold
- [x] Set freshness window
- [x] Connect Telegram
- [x] Receive personalized job notifications
- [x] Browse matching jobs in the dashboard

### Admin portal

- [x] Manage users
- [x] Manage companies
- [x] Approve or reject company requests
- [x] Manage watchlists
- [x] Connector health
- [x] Poll history
- [x] Notification history
- [x] Scheduler status
- [x] User preferences if needed

### Backend

- [x] PostgreSQL
- [x] Authentication
- [x] Role-based access
- [x] Configurable catalog
- [x] Job collectors
- [x] AI matching
- [x] Deduplication
- [x] Telegram notifications
- [x] User-specific filtering

## Intentionally out of scope for Version 1

These are valuable, but they belong in Version 2 or later:

- Resume tailoring
- Cover letter generation
- Recruiter discovery
- Company intelligence
- Application package
- ATS autofill
- Browser extension
- LinkedIn automation
- AI interview preparation
- Analytics based on months of usage

Delaying these keeps the build focused on a reliable daily job discovery loop first.

## Version 1 success metric

Do not measure success by feature count.

Use this question instead:

> Can I rely on this as my only job discovery tool for the next month?

That means:

- Important opportunities are not missed
- Notifications are relevant
- Setup is simple
- Preferences work
- Company requests are easy
- Admin operations are manageable

## Post-launch roadmap

Each version should build on the previous one instead of trying to solve everything at once.

1. Version 1.0: Job Discovery
2. Version 1.5: Resume Intelligence
3. Version 2.0: Application Copilot
4. Version 3.0: Networking Intelligence
5. Version 4.0: Career Intelligence Platform

## Pre-launch UX addition

Add a simple readiness checklist to the user dashboard so users can immediately see whether they are fully configured.

```text
Setup Progress
[x] Account created
[x] Resume uploaded
[x] Preferences configured
[x] Companies selected
[x] Telegram connected

Status:
Ready: You're actively receiving job alerts.
```

If something is missing, show a direct action hint:

```text
Needs attention: Resume not uploaded
Upload a resume to improve AI matching.
```

```text
Needs attention: Telegram not connected
Connect Telegram to receive instant notifications.
```

This should reduce setup confusion and support requests.

## Page design guardrails

Version 1 now has a dedicated UI page checklist in [UI Page Guardrails](ui-guardrails.md).

Use it for every new user or admin screen.

The short version:

- no duplicate page headers or repeated hero sections
- no edge-touching content inside cards
- real loading states are required
- large inventories must load in batches instead of rendering everything
- counts should come from aggregated payloads, not full-record fetches

This is now part of the Version 1 quality bar, not optional polish.

## Delivery rule

No new major features should start until Version 1 is deployed and used in production.

Finish authentication, Telegram onboarding, company approval flow, and deployment to Vercel plus Render first. Then use the product for a couple of weeks before committing to Version 2 scope.
